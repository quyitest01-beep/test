# 智能数据查询系统 - 开发任务计划

## 项目概述
本项目旨在构建一个基于云数据库的智能数据查询系统，解决以下核心场景问题：
1. 为非技术用户提供便捷的数据查询方案，使其无需掌握SQL技能即可通过数据库快速查询数据
2. 处理大规模数据分包导出excel，解决传统Excel数据量限制问题
3. 满足多样化查询需求，提升查询效率，避免纯人工处理带来的低效问题

### PDF服务和导出服务Content-Disposition编码修复 (已完成)
**任务目标**：修复PDF服务和导出服务中Content-Disposition响应头包含非ASCII字符导致的错误。

**问题分析**：
1. **错误现象**：`TypeError: [ERR_INVALID_CHAR]: Invalid character in header content ["Content-Disposition"]`
2. **根本原因**：HTTP响应头中的Content-Disposition字段包含中文等非ASCII字符
3. **影响范围**：
   - PDF服务的`/render`和`/render-url`端点
   - 导出服务的`/api/export/download/:filename`端点

**实现计划**：
- [x] 排查所有设置Content-Disposition响应头的位置
- [x] 修复PDF服务的文件名编码问题
- [x] 修复导出服务的文件名编码问题
- [x] 更新故障排查文档
- [x] 更新worklog.md记录修复过程

**当前进度**：
- ✅ 完成问题根因分析
- ✅ 完成PDF服务修复（`backend/pdf-service/server.js`）
- ✅ 完成导出服务修复（`backend/routes/export.js`）
- ✅ 完成故障排查文档更新
- ✅ 任务已完成

**修复方案**：
- 将所有非ASCII字符替换为下划线，创建ASCII安全的fallback文件名
- 对原始文件名进行URL编码，保留完整的中文字符信息
- 使用RFC 5987格式：`filename="${safeFilename}"; filename*=UTF-8''${encodedFilename}`

**验证方法**：
1. 重启PDF服务和主后端服务
2. 测试包含中文字符的PDF生成
3. 测试包含中文字符的文件下载
4. 确认不再出现`ERR_INVALID_CHAR`错误

**服务重启问题**：
- 修改代码后必须完全停止旧进程才能生效
- 查找占用端口的进程：`netstat -ano | findstr "8787 8000"`
- 强制终止进程：`taskkill /F /PID <进程ID>`
- 验证新进程已启动：检查PID是否改变

**最终状态**：✅ 代码修复完成，服务已成功重启并生效

```yaml
  LEFT JOIN games g ON r.game_id = g.id
```

```yaml
name: "业务指标报表生成"
description: "生成包含多个业务指标的定期报表"
parameters:
  - name: "metrics"
    type: "array"
    required: true
    values: ["MAU", "DAU", "retention", "net_revenue", "ARPU", "LTV"]
    aliases: ["月活", "日活", "留存", "净营收", "指标"]
  - name: "report_period"
    type: "enum"
    values: ["daily", "weekly", "monthly"]
    required: true
    aliases: ["日报", "周报", "月报"]
  - name: "auto_send"
    type: "boolean"
    default: false
    aliases: ["自动发送", "定期发送"]
text_patterns:
  - pattern: "(月活|日活|留存|净营收).*?(月报|周报|日报).*?发送"
    mapping:
      metrics: ["MAU", "DAU", "retention", "net_revenue"]
      report_period: "monthly"
      auto_send: true
query_templates:
  MAU: |
    SELECT COUNT(DISTINCT user_id) as mau
    FROM user_activities 
    WHERE DATE(created_at) >= DATE_SUB(CURDATE(), INTERVAL 30 DAY)
  DAU: |
    SELECT DATE(created_at) as date, COUNT(DISTINCT user_id) as dau
    FROM user_activities 
    WHERE DATE(created_at) >= DATE_SUB(CURDATE(), INTERVAL 7 DAY)
    GROUP BY DATE(created_at)
  retention: |
    SELECT 
      cohort_month,
      period_number,
      COUNT(DISTINCT user_id) / first_value(COUNT(DISTINCT user_id)) OVER (PARTITION BY cohort_month ORDER BY period_number) as retention_rate
    FROM user_cohort_analysis
    GROUP BY cohort_month, period_number
  net_revenue: |
    SELECT 
      DATE(created_at) as date,
      SUM(amount - cost) as net_revenue
    FROM revenue_records 
    WHERE DATE(created_at) >= DATE_SUB(CURDATE(), INTERVAL 30 DAY)
    GROUP BY DATE(created_at)
export_rules:
  format: "pdf"
  template: "business_metrics_template.html"
  filename: "${report_period}_metrics_report_${date}.pdf"
  auto_schedule:
    enabled: "${auto_send}"
    frequency: "${report_period}"
    recipients: ["analytics@company.com", "management@company.com"]
```

```json
      "enabled": true,
      "method": "email",
      "recipients": ["manager@company.com"],
      "subject": "${商户名称} ${月份} 游戏数据报告"
    }
```

### 用户交互流程设计

#### 用户交互流程设计

**1. 智能文本解析流程**
```
用户输入文案 → 文本预处理 → 参数提取 → 意图识别 → 模板匹配 → 参数验证 → 查询执行
```

**详细步骤**：
1. **文本预处理**：清理和标准化用户输入
2. **参数提取**：使用NLP引擎提取关键参数
   - 游戏参数：MerchantID、biz_id、round_id
   - 时间参数：时间范围表达式解析
   - 货币参数：货币代码识别
   - 业务指标：KPI指标识别
3. **意图识别**：判断用户查询意图
   - 异常检测查询
   - 时间范围数据查询
   - 业务指标报表生成
4. **模板匹配**：根据参数和意图匹配最佳模板
5. **查询执行**：执行数据库查询和处理逻辑

**2. 时间范围查询流程**
```
时间表达式 → 时间解析 → 日期范围确定 → 货币筛选 → 批量查询 → 数据聚合
```

**详细步骤**：
1. **时间表达式解析**：将"1号到31号"转换为具体日期
2. **相对时间处理**：处理"本月"、"上周"等相对时间
3. **货币类型识别**：识别PHP、INR等货币代码
4. **批量数据查询**：按时间范围和货币类型查询
5. **数据聚合统计**：按日期和货币分组统计

**3. 业务指标报表流程**
```
指标识别 → 报表周期确定 → 多指标查询 → 数据计算 → 报表生成 → 自动发送
```

**详细步骤**：
1. **业务指标识别**：识别MAU、DAU、留存、净营收等指标
2. **报表周期确定**：确定日报、周报、月报类型
3. **多指标并行查询**：同时执行多个指标的查询
4. **指标数据计算**：应用相应的计算公式
5. **报表模板渲染**：生成可视化报表
6. **自动化调度**：配置定期生成和发送

**4. Telegram Bot交互流程**
```
消息接收 → 用户识别 → 意图解析 → 参数检查 → 信息收集 → 需求确认 → 查询执行 → 结果推送
```

**详细步骤**：
1. **消息接收**：监听TG群组中@机器人的消息
2. **用户识别**：验证用户身份和群组权限
3. **意图解析**：调用智能文本解析识别查询意图
4. **参数检查**：验证查询参数的完整性
5. **信息收集**：缺失参数时主动询问用户
   - "请提供具体的时间范围"
   - "需要查询哪些货币类型？"
   - "请确认MerchantID和biz_id"
6. **需求确认**：向用户确认最终查询需求
   - "确认查询1-31号的PHP和INR数据，对吗？"
7. **查询执行**：调用后端智能查询系统
8. **进度反馈**：实时推送查询进度状态
   - "正在查询数据...⏳"
   - "数据处理中，预计还需2分钟...📊"
9. **结果推送**：发送查询结果和文件到群组
   - 文本格式的摘要结果
   - Excel/PDF文件下载链接
   - 可视化图表图片
10. **异常处理**：查询失败时的错误提示和重试机制

**4. 场景模板选择流程**
```
用户登录 → 浏览模板库 → 选择场景模板 → 查看模板详情 → 确认使用
```

**5. 参数配置流程**
```
模板加载 → 显示参数表单 → 用户填写参数 → 参数验证 → 预览查询 → 确认配置
```

**6. 执行监控流程**
```
开始执行 → 实时进度显示 → 状态更新推送 → 异常处理 → 结果展示 → 操作完成
```

**7. 异常检测分析流程**
```
数据查询 → 规则引擎检测 → 统计分析 → 异常等级评估 → 原因分析 → 结果可视化
```

**详细步骤**：
1. **异常规则匹配**：应用预设的异常检测规则
2. **异常评分**：计算异常程度评分
3. **风险等级判定**：根据评分确定风险等级
4. **处理建议生成**：提供相应的处理建议

#### 主要操作流程
1. **选择场景模板**
   - 用户从预设模板列表中选择
   - 或创建新的自定义模板
   - 预览模板配置和参数要求

   - 实时显示执行进度和状态

5. **结果查看和管理**
   - 查看执行结果和生成文件
   - 下载导出文件
   - 查看执行日志和错误信息
