# 数据库负值存储说明

## 问题

数据库可以存储负值吗？这种也算是数值吗？

## 答案

**是的，数据库可以存储负值，负值也是数值类型，可以正常进行数值计算和比较。**

## 数值类型存储负值能力

### 1. DECIMAL 类型（支持负值）✅

```sql
-- DECIMAL 默认支持正负值
`total_ggr` DECIMAL(20, 4)  -- 可以存储：-100.50, 0, 100.50

-- 如果需要只允许正值，需要添加 CHECK 约束
`total_ggr` DECIMAL(20, 4) CHECK (`total_ggr` >= 0)  -- 只允许 >= 0
```

**示例**：
```sql
-- 可以存储负值
INSERT INTO revenue_data (total_ggr) VALUES (-100.50);  -- ✅ 可以
INSERT INTO revenue_data (total_ggr) VALUES (100.50);   -- ✅ 可以
INSERT INTO revenue_data (total_ggr) VALUES (0);        -- ✅ 可以
```

### 2. INT UNSIGNED 类型（不支持负值）❌

```sql
-- UNSIGNED 表示无符号，不支持负值
`bet_users` INT UNSIGNED  -- 只能存储：0 到 4294967295

-- 如果尝试存储负值会报错
INSERT INTO game_user_data (bet_users) VALUES (-10);  -- ❌ 错误！
```

### 3. INT 类型（支持负值）✅

```sql
-- INT 支持正负值
`bet_users` INT  -- 可以存储：-2147483648 到 2147483647
```

## 业务场景分析

### 1. 可能需要负值的字段

#### GGR（Gross Gaming Revenue）
- **场景**：当派奖金额大于投注金额时，GGR为负数
- **示例**：投注 $100，派奖 $150，GGR = -$50
- **存储**：`DECIMAL(20, 4)` ✅ 支持负值

#### RTP（Return to Player）
- **场景**：理论上可能超过100%（派奖 > 投注）
- **示例**：RTP = 150%（玩家赢了）
- **存储**：`DECIMAL(10, 4)` ✅ 支持负值和超过100的值

#### 投注金额、派奖金额
- **场景**：通常应该是正数
- **建议**：可以使用 CHECK 约束限制为 >= 0

### 2. 不应该为负值的字段

#### 用户数（bet_users, daily_users等）
- **场景**：用户数不应该是负数
- **存储**：`INT UNSIGNED` ✅ 自动限制为非负数

#### 留存率（retention_rate）
- **场景**：留存率应该在 0-100% 之间
- **存储**：`DECIMAL(10, 4)` 或添加 CHECK 约束

#### 总局数（total_rounds）
- **场景**：局数不应该是负数
- **存储**：`BIGINT UNSIGNED` ✅ 自动限制为非负数

## 当前表结构检查

### revenue_data 表

```sql
`total_bet` DECIMAL(20, 4)      -- ✅ 支持负值（但业务上可能不需要）
`total_payout` DECIMAL(20, 4)   -- ✅ 支持负值（但业务上可能不需要）
`total_ggr` DECIMAL(20, 4)      -- ✅ 支持负值（业务需要：可能为负）
`rtp` DECIMAL(10, 4)            -- ✅ 支持负值（业务需要：可能超过100%）
`total_rounds` BIGINT UNSIGNED  -- ❌ 不支持负值（正确：局数不能为负）
```

**建议**：
- `total_ggr`：保持支持负值 ✅（业务需要）
- `rtp`：保持支持负值或超过100% ✅（业务需要）
- `total_bet`, `total_payout`：可以考虑添加 CHECK 约束限制为 >= 0

### game_user_data 表

```sql
`bet_users` INT UNSIGNED  -- ❌ 不支持负值（正确：用户数不能为负）
```

**建议**：保持 `INT UNSIGNED` ✅（用户数不应该为负）

### game_retention_data 表

```sql
`daily_users` INT UNSIGNED           -- ❌ 不支持负值（正确）
`d1_users` INT UNSIGNED              -- ❌ 不支持负值（正确）
`d1_retention_rate` DECIMAL(10, 4)   -- ✅ 支持负值（但不应该为负）
```

**建议**：
- 用户数字段：保持 `INT UNSIGNED` ✅
- 留存率字段：可以考虑添加 CHECK 约束限制为 0-100

## 优化建议

### 方案1：添加 CHECK 约束（推荐）

```sql
-- 限制 GGR 可以为负（业务需要）
`total_ggr` DECIMAL(20, 4)  -- 保持现状，允许负值

-- 限制投注和派奖必须 >= 0
`total_bet` DECIMAL(20, 4) CHECK (`total_bet` >= 0),
`total_payout` DECIMAL(20, 4) CHECK (`total_payout` >= 0),

-- 限制留存率在 0-100 之间
`d1_retention_rate` DECIMAL(10, 4) CHECK (`d1_retention_rate` >= 0 AND `d1_retention_rate` <= 100),
```

### 方案2：使用 UNSIGNED（仅适用于不应该为负的字段）

```sql
-- 用户数：使用 UNSIGNED
`bet_users` INT UNSIGNED  -- 自动限制为 >= 0

-- 局数：使用 UNSIGNED
`total_rounds` BIGINT UNSIGNED  -- 自动限制为 >= 0
```

## 实际业务场景

### GGR 为负值的情况

```sql
-- 场景：玩家赢了，派奖 > 投注
INSERT INTO revenue_data (total_bet, total_payout, total_ggr)
VALUES (100.00, 150.00, -50.00);  -- GGR 为 -$50

-- 查询负GGR的数据
SELECT * FROM revenue_data WHERE total_ggr < 0;
```

### RTP 超过 100% 的情况

```sql
-- 场景：派奖率超过100%
INSERT INTO revenue_data (total_bet, total_payout, rtp)
VALUES (100.00, 150.00, 150.00);  -- RTP = 150%

-- 查询高RTP的数据
SELECT * FROM revenue_data WHERE rtp > 100;
```

## 总结

1. **数据库可以存储负值**：
   - `DECIMAL`、`INT` 等类型支持负值
   - `INT UNSIGNED`、`BIGINT UNSIGNED` 不支持负值

2. **业务字段建议**：
   - **GGR**：允许负值 ✅（业务需要）
   - **RTP**：允许负值或超过100% ✅（业务需要）
   - **用户数**：不允许负值 ✅（使用 UNSIGNED）
   - **留存率**：限制在 0-100 之间 ✅（使用 CHECK 约束）

3. **当前表结构**：
   - ✅ 已经正确使用了 `INT UNSIGNED` 限制用户数为非负数
   - ✅ `DECIMAL` 类型支持负值，满足GGR和RTP的业务需求
   - 💡 可以考虑添加 CHECK 约束进一步限制数据范围
