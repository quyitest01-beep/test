# 新游戏评级数据处理器使用说明

## 功能概述

`game-rating-data-processor.js` 是一个 n8n Code 节点，用于将上游数据（Merge2.json 格式）处理成便于 AI 分析的结构化数据，用于生成新游戏评级报告。

## 输入数据格式

输入数据应该是一个数组，包含以下三个部分：

### 1. 投注用户数数据
- **表头**：`["日期", "游戏名", "投注用户数"]`
- **数据行**：包含日期、游戏名、投注用户数的记录
- **特殊行**：游戏名为"合计"或"total"的行表示全平台汇总数据

### 2. 营收数据
- **表头**：`["时间", "商户名", "货币", "USD汇率", "游戏名", "总投注", "总投注USD", "总派奖", "总派奖USD", "总局数", "RTP", "GGR", "GGR-USD"]`
- **数据行**：包含各商户、各游戏的营收明细
- **特殊行**：游戏名为"合计"或"total"的行表示全平台汇总数据

### 3. 留存数据
- **无明确表头**（从数据内容推断）
- **数据行格式**：`[游戏名, 商户名, 时间戳, 用户类型, 当日用户数, D1留存, D2留存, ..., D14留存]`
- **用户类型**：`"新用户留存"` 或 `"活跃用户留存"`
- **过滤规则**：只统计当日用户数 >= 50 的记录

## 评级标准

处理器输出的数据包含完整的评级标准参考，AI 可以根据以下标准进行评级：

### 100分 - 首页第一位
- 下注人数全平台占比 > 10%
- 留存占比 > 30%
- 下注金额全平台占比 > 10%
- 游戏人均GGR > 40

### 80分 - 首页2-6位
- 下注人数全平台占比：5%-9%
- 留存占比：25-29%
- 下注金额全平台占比：5%-9%
- 游戏人均GGR：25-39

### 60分 - 分组页1-6位
- 下注人数全平台占比：3%-4%
- 留存占比：21-24%
- 下注金额全平台占比：3%-4%
- 游戏人均GGR：15-24

### 40分 - 分组页7-12位
- 下注人数全平台占比：1%-2%
- 留存占比：15-20%
- 下注金额全平台占比：1%-2%
- 游戏人均GGR：5-14

### 20分 - 分组页13位以后
- 下注人数全平台占比 < 1%
- 留存占比 < 14%
- 下注金额全平台占比 < 1%
- 游戏人均GGR < 5

## 指标权重

- **人数**：0.2
- **人均下注金额**：0.3
- **人均GGR**：0.2
- **留存**：0.3

## 使用步骤

1. **在 n8n 工作流中添加 Code 节点**
2. **将上游数据连接到 Code 节点**
3. **将 `game-rating-data-processor.js` 的代码复制到 Code 节点中**
4. **运行工作流，查看输出数据**
5. **将输出数据传递给 AI 节点进行评级分析**

## 注意事项

1. **全平台数据**：
   - 处理器会尝试从输入数据中查找"合计"行来获取全平台数据
   - 如果没有找到"合计"行，会尝试汇总所有数据（可能不准确）
   - 建议确保输入数据中包含全平台汇总行

2. **留存数据过滤**：
   - 只统计当日用户数 >= 50 的留存记录
   - 这可以过滤掉样本量过小的数据，提高准确性

3. **数据周期**：
   - 处理器会自动从用户数据中提取日期范围
   - 如果无法提取，dataPeriod 字段会显示"未知"

4. **游戏名识别**：
   - 处理器会自动识别目标游戏名（从第一个非表头数据项中提取）
   - 确保输入数据中只包含一个游戏的数据，或者明确标识目标游戏

## 故障排查

### 问题1：全平台数据为 0 或 null
- **原因**：输入数据中缺少"合计"行
- **解决方案**：确保上游数据包含全平台汇总数据，或者在上游添加汇总节点

### 问题2：留存率为 0
- **原因**：留存数据中所有记录的当日用户数都 < 50，被过滤掉了
- **解决方案**：检查留存数据，确认是否有足够的样本量

### 问题3：游戏名识别错误
- **原因**：输入数据中包含多个游戏的数据
- **解决方案**：确保输入数据只包含目标游戏的数据，或修改代码以明确指定游戏名
